# DATA_MODEL.md

MongoDB collections, document shapes, indexes, and conventions. Source of truth for what the database holds.

---

## Database

- **Name**: `todoapp` (configurable via `MONGODB_DB` env var)
- **Engine**: MongoDB Atlas (cloud), accessed via Motor (async driver)
- **Two collections**: `users`, `todos`

---

## Collection: `users`

One document per registered user.

### Document shape

```jsonc
{
  "_id": ObjectId,           // primary key
  "name": "Pankaj W.",       // 1–80 chars
  "email": "user@example.com", // lowercased, unique
  "password_hash": "$2b$...", // bcrypt hash (NEVER plaintext)
  "created_at": ISODate
}
```

### Constraints

| Field | Type | Required | Notes |
|---|---|---|---|
| `_id` | ObjectId | yes | Auto-generated |
| `name` | string | yes | 1–80 chars after trim |
| `email` | string | yes | Lowercased on insert, validated via `pydantic.EmailStr` |
| `password_hash` | string | yes | bcrypt, generated by `bcrypt.hashpw(...)` |
| `created_at` | datetime (UTC) | yes | Set server-side at registration |

### Indexes

```js
db.users.createIndex({ email: 1 }, { unique: true })
```

The unique constraint on `email` is how we prevent duplicate accounts. The lowercased-on-insert convention means `User@Example.com` and `user@example.com` collide.

### API exposure

Public response (`UserOut`) drops `password_hash` and renames `_id` → `id: str`.

```ts
type User = {
  id: string;        // hex of _id
  name: string;
  email: string;
  created_at: string; // ISO 8601 UTC
}
```

## Collection: `todos`

One document per todo. Recurring todos are a single document whose `due_date` advances on completion (not a series of records).

### Document shape

```jsonc
{
  "_id": ObjectId,
  "user_id": ObjectId,            // ref to users._id — owner
  "title": "Pay rent",            // 1–200 chars
  "description": "Via bank...",   // optional, ≤ 2000 chars
  "status": "pending",            // pending | in_progress | completed
  "priority": "high",             // low | medium | high
  "due_date": ISODate,            // optional — when it's due
  "recurrence": "monthly",        // optional — daily | weekly | monthly | null
  "last_completed_at": ISODate,   // optional — set when a recurring task was last completed
  "created_at": ISODate,
  "updated_at": ISODate
}
```

### Constraints

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `_id` | ObjectId | yes | auto | |
| `user_id` | ObjectId | yes | — | Must match `users._id` of the JWT bearer |
| `title` | string | yes | — | 1–200 chars, trimmed |
| `description` | string \| null | no | null | ≤ 2000 chars |
| `status` | enum string | yes | `"pending"` | `pending` \| `in_progress` \| `completed` |
| `priority` | enum string | yes | `"medium"` | `low` \| `medium` \| `high` |
| `due_date` | datetime (UTC) \| null | no | null | Stored as UTC. Mobile converts to local for display. |
| `recurrence` | enum string \| null | no | null | `daily` \| `weekly` \| `monthly` \| null |
| `last_completed_at` | datetime (UTC) \| null | no | null | Set by the bump-in-place flow when a recurring task is completed |
| `created_at` | datetime (UTC) | yes | now() | |
| `updated_at` | datetime (UTC) | yes | now() | Bumped on every update |

### Indexes

```js
db.todos.createIndex({ user_id: 1, created_at: -1 })
db.todos.createIndex({ user_id: 1, due_date: 1 })
db.todos.createIndex({ user_id: 1, status: 1 })
db.todos.createIndex({ user_id: 1, priority: 1 })

// For text search on title + description (case-insensitive substring matching
// currently uses $regex, which scans. If volume grows, switch to text index):
db.todos.createIndex({ title: "text", description: "text" })
```

The `(user_id, ...)` compound indexes are mandatory — every query in `todo_service.py` filters by `user_id` first. Without that lead column, the planner does a collection scan.

### API exposure

Public response (`TodoOut`) renames `_id` → `id: str`, stringifies `user_id`, and serializes datetimes as ISO 8601 with explicit UTC offset.

```ts
type Todo = {
  id: string;
  user_id: string;
  title: string;
  description?: string | null;
  status: "pending" | "in_progress" | "completed";
  priority: "low" | "medium" | "high";
  due_date?: string | null;
  recurrence?: "daily" | "weekly" | "monthly" | null;
  last_completed_at?: string | null;
  created_at: string;
  updated_at: string;
}
```

## Datetime convention

All datetimes in the DB are **UTC-aware**. The backend forces this on read via `_as_utc(...)` in `todo_service.py` — naive datetimes from MongoDB get tagged as UTC before serialization.

Why this matters: JavaScript's `new Date("2025-05-22T10:00:00")` (no Z, no offset) parses as local time on iOS but UTC on web — silent corruption. Explicit `Z` or offset everywhere prevents this.

**Mobile responsibility**: when the user picks a local time, the mobile client must convert to UTC before sending. When displaying, convert UTC back to local. See `mobile/src/utils/` for the helpers.

**Legacy fix**: tasks created by old versions of AI Chat saved local wall-clock time as if it were UTC. The `POST /todos/{id}/repair-timezone` endpoint subtracts an offset to correct them. Remove this endpoint after the affected records are migrated out.

## Recurrence semantics

A recurring todo is **one document**, not many.

**On completion** (`status` transition → `completed` AND `recurrence != null`):
1. `status` is forced back to `pending` (not actually marked completed).
2. `due_date` advances:
   - `daily` → `+1 day`
   - `weekly` → `+7 days`
   - `monthly` → `+1 month`, day clamped to last valid day (Jan 31 → Feb 28/29)
3. `last_completed_at` is set to `now()`.

**On uncomplete** (`POST /todos/{id}/uncomplete`):
1. `due_date` walks back one interval (same clamping rules in reverse).
2. `last_completed_at` is cleared.
3. `status` is set to `pending`.

This gives "no completion history" semantics — the user always sees the next occurrence, never a backlog of past instances.

## ID strategy

- Mongo `_id` is `ObjectId` (default).
- API never exposes `ObjectId` — always converts to its 24-char hex string and labels the field `id`.
- API path params accepting an id (`/todos/{id}`) parse with `ObjectId(id)` inside a try/except → 400 on `InvalidId`, then 404 on missing-or-not-mine.

---

## Auth model

- JWT is stateless. No `sessions` collection.
- JWT `sub` claim contains the user's `_id` as a hex string.
- `app/core/deps.py:get_current_user` decodes the JWT, parses the sub as ObjectId, and loads the user document. If any step fails → 401.
- No refresh tokens. Token expires in 7 days; user logs in again.
- No revocation. To revoke all tokens, rotate `JWT_SECRET` (invalidates every existing token).

## Future considerations (not implemented)

- **Subtasks as documents** — currently the AI parses subtasks into strings; if they become first-class todos, add a `parent_id: ObjectId` field and a `(user_id, parent_id)` index.
- **Tags / labels** — add a `tags: string[]` field and a `(user_id, tags)` multikey index.
- **Soft delete** — add `deleted_at: ISODate | null` and filter it out of every query. Adds complexity; current hard-delete is fine for the scale.
- **Text search at scale** — replace `$regex` filtering with the existing `text` index + `$text` operator once collection exceeds ~10k docs per user.
